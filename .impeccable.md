## Design Context

### Users
Server administrators and developers deploying self-hosted auth infrastructure. They're technical, time-pressed, managing auth for their own products. Dashboard used in focused work sessions — configuring OAuth providers, debugging login failures, managing API keys, reviewing audit logs. Most usage on desktop in dark environments (office, home desk, late nights). They chose SharkAuth because they want control, not hand-holding.

### Brand Personality
**Bold. Elegant. Fast.**

"No bullshit" — the README says it, the product lives it. Single binary, no vendor lock-in, no pricing tiers for basic features. The dashboard should feel like a precision instrument: sharp, confident, information-dense. Every pixel earns its place.

### Aesthetic Direction
- **Tone:** High-density command center. Linear is the north star — compact, keyboard-driven, no wasted space. Anti-reference: Railway (too rounded, too playful, too much padding).
- **Theme:** Dark-only. The shark logo is white on black — the entire brand was born in dark mode. Light text on dark surfaces, oklch semantic colors for status.
- **Palette:** B&W foundation with oklch semantic accents. No blue branding — the current neutral palette is final. Surfaces step from `#000` through `#1c1c1c`. Accent colors reserved for meaning: success/warn/danger/agent.
- **Density:** Compact by default (13px base, 28px buttons, 44px topbar). Information density over whitespace. Every row, every card, every metric should communicate.
- **Motion:** 60-120ms transitions. No bounce, no elastic. Objects decelerate smoothly.

### Component Shape Philosophy (locked 2026-04-24)
**Square over round.** Border radius scale capped low: `3-6px` only. Never `8+`, never pill, never circle (except status dots and avatars).

- **Reference pages (gold standard):** `users.tsx`, `agents_manage.tsx`, `sessions.tsx` — copy their density, radius, padding scale, row heights.
- **Anti-reference pages (rebuild needed):** `proxy_wizard.tsx`, `proxy_config.tsx`, `get_started.tsx`, `Walkthrough.tsx` — too round, too big, too much padding. Components feel templated. **Avoid this aesthetic at all costs.**
- **Right-side detail panel (loved pattern):** `position: fixed; top: 0; right: 0; bottom: 0` overlay drawer for inspecting a row. Square edges, hairline border on left, 380-440px wide. Replaces modals for inspection/edit flows. Use this pattern wherever a row reveals deeper detail.
- **Sizes:** Cards radius 4-5px. Inputs radius 4px. Buttons radius 4-5px. Badges/pills radius 3px. Avatars/status dots circular (only exception).
- **Padding:** 8-16px inside cards. 10-12px row padding. Never `padding: 24px` for chrome.

### Typography Direction
- **Display:** Hanken Grotesk — geometric, sharp terminals, commanding in bold weights. Fits "bold, elegant, fast."
- **Body:** Manrope — geometric with slightly rounded terminals, high x-height, excellent dark-bg readability at 13px. Subtle contrast with Hanken without clashing.
- **Mono:** Azeret Mono — geometric consistency with Hanken, clean at small sizes for IDs/tokens/keys, not overused.
- **Scale:** Fixed rem scale (no fluid type — this is product UI, not marketing). Fewer sizes, more contrast between them. At least 1.25 ratio between steps.

### Design Principles
1. **Simple until you need deep.** Everything looks straightforward on first glance. Complexity exists but stays behind one click, one expand, one hover. Progressive disclosure everywhere.
2. **Density over decoration.** Show more data, not more chrome. Every pixel of padding must justify itself.
3. **Square over round.** Sharp corners, hairline borders, low radius. Round = playful = wrong brand.
4. **Meaning through restraint.** Color = status. If everything is colorful, nothing communicates. B&W default, color for signals.
5. **Speed as UX.** Optimistic mutations, skeleton loading, keyboard shortcuts. The fastest dashboard interaction is one that doesn't require waiting.
6. **Act where you see.** Every data point that can be acted on has its action inline. Right-panel drawer over modal navigation.
7. **Control without complexity.** Admin should feel in control — not overwhelmed. Obvious defaults, clear labels, no jargon where plain words work.
