## Problem Statement

Every MCP server shipping today has zero identity layer. Teams hand-roll JWT checks, hard-code API keys, or punt on auth entirely. Auth0 and Clerk were built for humans logging into SaaS — not agents making delegated calls on a user's behalf. Auth0 charges 50% uplift for agent auth ($53+/mo). Okta Agent Gateway is enterprise-plan-only with undisclosed pricing. Stytch ships MCP compliance but treats agents as M2M clients, not first-class identities with delegation chains.

The broader 2026 auth landscape is hostile to builders: Auth0 is expensive and closed, Clerk gets steep at scale, WorkOS is narrow (SAML-only), Okta is enterprise-locked. Every OSS project, every compliance-heavy mid-market, every agent-native startup faces the same forced choice: pay a SaaS vendor and route user credentials through their cloud, or hand-roll OAuth and get security wrong.

SharkAuth is the third option: a single-binary, open-source OAuth 2.1 authorization server with the identity primitives customers already need (SSO, passkeys, MFA, magic links, orgs, RBAC) plus native RFC 8693 agent delegation chains, plus a reverse-proxy mode that drops in front of any app with zero code changes.

## Status Quo

What MCP devs are doing right now to solve this:

1. **Hand-rolled JWT middleware** — shared-secret bearer tokens, no delegation, no rotation, no audit. Breaks the moment an agent needs to call another agent's resource.
2. **Pay Auth0/Clerk** — adds $50-200/mo per environment, ships human-centric flows agents don't use, routes user tokens through third-party cloud.
3. **Punt entirely** — MCP server has no auth; protected only by being behind a VPN or corporate network. Common in internal tooling, catastrophic when exposed.

The cost: either real dollars (SaaS), real engineering hours (hand-rolled), or real security debt (punted). All three worsen as MCP fan-out increases — one user connecting to 10 MCP servers × 3 environments = 30 auth decisions.

## Target User & Narrowest Wedge

**Primary target (wedge):** Open-source MCP server authors + indie devs shipping agent-facing products. Solo or small-team. Already running Go/Node/Python services. Already hate Auth0 pricing. Already nervous about sending user tokens through third-party clouds. TypeScript + Python-leaning.

**Secondary target (expansion):** Mid-market B2B SaaS with compliance needs (healthcare, fintech, EU residency) who rejected Clerk/Auth0 for cloud-lock-in reasons and are currently hand-rolling auth on Keycloak or Ory.

**Narrowest wedge (first 30 days):** "Drop SharkAuth in front of your MCP server — get OAuth 2.1 + DCR + DPoP + agent delegation in 5 minutes, zero code changes, one binary." Ship that. Nothing else.

**Tertiary expansion (post-YC, 12 months):** Hosted SharkAuth Cloud as Supabase-style self-host-or-hosted choice. Blocked today by SQLite-only database (see Open Questions).

## Approaches Considered

### Approach A: YC-First (ship minimum, apply in 10 days)

**Summary:** Freeze scope at current feature matrix. Spend 10 days on (a) 3 design-partner installs, (b) 5-min demo video, (c) OpenAPI spec + GitHub Releases binary publish, (d) YC app. Ship v1.0 launch post-YC-interview.

**Effort:** M. **Risk:** Med (traction risk dominates — must land 3 partners).

**Pros:**
- Hits YC deadline with real evidence pack (shipped code + partners + demo).
- Forces wedge discipline — every hour goes to the narrowest value story.
- Partner calls during YC interview reinforce category pull.

**Cons:**
- Means delaying SAML IdP, Postgres, hosted cloud, passkey polish.
- If 3 partners don't land, YC app is weakened regardless.
- Solo founder at 18 with no revenue remains structural risk.

**Reuses:** Entire existing codebase unchanged. Matrix above is the product story.

---

### Approach B: Product-First (polish + launch, apply next batch)

**Summary:** Skip YC P26. Spend 90 days shipping SAML IdP, Postgres, hosted SharkAuth Cloud beta, polished docs site, 10-person design-partner cohort. Apply to YC next batch (W27) from a position of real traction.

**Effort:** XL. **Risk:** Med-High (category timing risk — Auth0 ships half-baked MCP mode, Stytch self-host tier, Cloudflare announces something, window closes).

**Pros:**
- Larger demand evidence pack by application time.
- Postgres + hosted cloud unlocks mid-market segment.
- Co-founder recruiting has more time to land.

**Cons:**
- 6-month delay. Category could commoditize fast.
- Solo-founder risk identical; extra time doesn't fix structure.
- Opportunity cost of not being in P26 batch network.

**Reuses:** Current codebase + roadmap. All audit items become deliverables.

---

### Approach C: Lateral — Open-source flagship + Cloud waiting list (YC-aware, partner-focused)

**Summary:** Treat OSS adoption as the traction metric, not paying customers. Blast launch: HN front page, Product Hunt, r/selfhosted, YC co-founder matching board. Goal in 10 days: 500 GitHub stars + 15 live `shark serve` installs + cloud waitlist of 200. Apply to YC with "category flagship" framing — not "we have revenue", but "we own the developer narrative in MCP auth."

**Effort:** M (launch effort concentrated in days 3-7). **Risk:** High-reward, High-risk — if launch lands, it carries the app; if it flops (weekday, HN algo, competing launch), back to Approach A without the polish buffer.

**Pros:**
- Matches Shark's actual leverage point (OSS, self-host, single binary).
- Star count + install count = quantifiable demand signal YC partners respect for infra.
- Category-owning narrative is defensible against Stytch/Clerk even pre-revenue.

**Cons:**
- OSS stars are a weaker signal than revenue — partners will still ask about monetization path.
- High execution cost if solo — launch day is 16 hours of responding to HN comments, Discord, GitHub issues.
- Burns marketing powder before polish is complete.

**Reuses:** Current codebase. Adds: landing page polish, launch copy, Discord server, HN/PH submission checklist.

## Recommended Approach

**Approach A with one Approach C element bolted on: launch publicly on Day 7 (HN + r/selfhosted) to manufacture traction signal in time for Day 10 YC submission.**

Reasoning:
- Approach A alone gives YC a technical story with weak traction. Pure traction risk.
- Approach B forfeits the P26 window + timing advantage for +6 months of polish. Trade rejected — category moves faster than 6 months.
- Approach A + mini-launch (Day 7) converts 48 hours of launch-day signal into the app's traction section. Worst case: no HN traction → submit app with 3 design partners + demo + technical story (still better than most pre-launch solo apps). Best case: HN front page → 500 stars + 15 installs + cloud waitlist = strong partner-interview story.
- Does not add scope. Adds one day of launch prep + one day of launch execution to Approach A's calendar.

## Open Questions

2. **Is the MIT-or-similar OSS license locked in?** Matters for partner due diligence and YC app. `LICENSE` file should be visible at root before launch.

5. **Co-founder search?** YC accepts solos but bar is higher. Worth one Day-1 post on YC co-founder matching board and one Monterrey-ecosystem ping. Don't let search block launch.
