## Purpose
ECDSA P-256 keypair generation, PEM encode/decode, KID derivation, and RFC 7517 JWK serialization for ES256 signatures. Prepared infrastructure — not currently wired into the active issuance path (RS256 is the default).

## Key types / functions
- `GenerateES256Keypair` (func, line 15) — ECDSA P-256 keypair via `crypto/rand`.
- `MarshalES256PrivateKeyPEM` (func, line 24) — PKCS#8 PEM.
- `ParseES256PrivateKeyPEM` (func, line 34) — inverse parse, type-asserts `*ecdsa.PrivateKey`.
- `MarshalES256PublicKeyPEM` (func, line 51) — PKIX PEM.
- `ComputeES256KID` (func, line 62) — base64url(SHA-256(DER))[:16] — same shape as RSA KID.
- `ES256PublicJWK` (func, line 74) — RFC 7517 EC JWK map: `{kty:EC, use:sig, alg:ES256, kid, crv:P-256, x, y}`; left-pads X/Y to exactly 32 bytes for spec compliance.

## Imports of note
- `crypto/ecdsa`, `crypto/elliptic` — P-256 curve.
- `crypto/x509`, `encoding/pem` — DER/PEM.

## Used by
- Not yet wired in production. Call sites would mirror `keys.go` once ES256 is added to the signing manager.

## Notes
- Symmetric API surface to `keys.go` so swapping algorithms is mostly registration work.
- Coordinate left-padding is mandatory: P-256 requires exactly 32-byte X/Y in JWK format; raw `big.Int.Bytes()` may strip leading zeros.
- KID derivation matches RSA's pattern so a JWKS endpoint can serve both algorithms uniformly.
