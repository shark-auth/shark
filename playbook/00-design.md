## Problem Statement

SharkAuth is a 30MB single-binary auth server with RFC-correct agent-native primitives (DPoP RFC 9449, token exchange RFC 8693, MCP-native OAuth 2.1, agent-aware audit). Backend is complete. The launch on Monday is at risk because a day-1 user cannot SEE the moat in 30 seconds. The dashboard renders agents as generic OAuth clients (client_id + scopes table — indistinguishable from Auth0). The Python SDK is ~60% of backend capability — missing the 5 helpers needed for a 10-line "Auth0 cannot match this" code snippet. The launch needs one viral artifact (visible moat demo) and one technical credibility artifact (SDK killer snippet), shipped within ~24h.

## Demand Evidence

Zero users have run the binary. One LinkedIn exchange with Hang Huang (InsForge CEO, YC P26): "love this a lot," "agent auth is a real pain point," "send launch posts, will engage." This is engagement promise + category validation. Not demand. Founder has not yet sat behind anyone using shark.

This is a known risk and is explicitly accepted as Monday's launch posture. Demand evidence will be generated DURING and AFTER launch, not before. Mitigation: ship a demo so compelling that one named integrator emerges from launch traffic.

## Status Quo

What MCP server builders / agent framework authors do today:
- Roll their own OAuth 2.1 flows (Anthropic MCP spec is recent, libraries lag)
- Bolt OAuth onto Auth0/Clerk/WorkOS — but those treat agents as machine clients with no native delegation primitive, no DPoP UI, no `act` claim awareness
- Skip auth entirely in dev, panic when shipping to production
- Hand-roll DPoP proof signing because RFC 9449 is fresh and SDK support is patchy

## Target User & Narrowest Wedge

**Target user (day-1):** MCP server builder or agent-framework maintainer who already understands OAuth and is currently hand-rolling DPoP + delegation. Specifically: people active in Anthropic MCP Discord, Cloudflare Agents team, Smithery contributors, Vercel AI SDK contributors, LangChain/LlamaIndex maintainers. They have the technical taste to recognize the moat in 30 seconds.

**Narrowest wedge:** "Drop-in agent auth for MCP servers." Single command to spin up auth + working delegation chain + DPoP-bound tokens. The killer artifact is a 30-second screencast showing 3 agents in a delegation chain with cryptographic proofs visible.

## Premises (agreed during session)

1. **Real moat = DPoP + delegation chain + audit, bound together visibly.** Any one in isolation is competitor parity. The trio together is unique.
2. **Day-1 user value = SEEING the moat in 30 seconds, not extending it.** Backend is done; exposure is the bottleneck.
3. **SDK ergonomics for launch > more backend features.** A 10-line snippet that Auth0 cannot replicate is more valuable than 5 new endpoints.
4. **One named integrator > three more shipped features.** Launch outcome quality is measured by "named user with working integration," not by feature count.

## Open Questions

1. **Which language SDK ships first with the killer snippet — Python or TypeScript?** Python has the strongest current foundation per explorer report. TypeScript is more visible to MCP server builders (most MCP servers are TS). Recommendation: ship Python first because it's closer to done, then port to TS in W18 with a "now in TS too" follow-up post.
2. **Should the demo HTML be styled monochrome to match dashboard `.impeccable.md v3` lock?** Yes — consistency matters for brand. Reuse dashboard tokens.
3. **Where does the demo HTML live after launch?** Self-host on `demo.sharkauth.dev` or just keep it as a generated artifact users run locally? Recommendation: both — local artifact for trust, hosted demo for shareability.
4. **Does the launch post explicitly mention the bounce points are deferred?** Yes — frame as "CLI/SDK first by design." Honest scoping wins more credibility than hidden bugs.
