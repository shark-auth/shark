# Demo App Plan — "Briefcase"

> **Status:** PLAN ONLY — no code. Solo dev implements post-YC-app submission (or in parallel).
> **Written:** 2026-04-27. YC application deadline 8 days out.

---

## Premise

Briefcase is an AI research assistant that reads your inbox and GitHub notifications, surfaces
what matters daily, and lets you spin up named agents to act on that context — drafting replies,
opening issues, summarising PR threads. A human logs in, connects Gmail + GitHub, and gets a
personalised morning briefing generated by a Claude-powered orchestrator that delegates to
narrow sub-agents for each vault provider. Every action is auditable, every credential is
DPoP-bound, every agent can be revoked in one click. It is the kind of tool a YC partner
running 40 portfolio companies would actually open every morning — not because it's clever,
but because it saves 45 minutes.

---

## Why this app, not another

- **Delegation is natural, not contrived.** "Fetch my Gmail summary" and "fetch my GitHub
  notifications" are two separate vault calls that naturally belong to two separate sub-agents
  delegated by one orchestrator. The `act` chain is architecturally motivated, not stapled on.

- **Both vault providers are verified (Gmail + GitHub).** No mocking required. Real OAuth
  round-trips, real provider responses, real token encryption at rest.

- **A notes app has no agents.** The only Shark-differentiating surface is human auth. Briefcase
  requires DPoP, token exchange, vault, cascade revoke — the full stack — to do its job.

- **Real daily utility.** A YC partner can actually use Briefcase to check their portfolio
  updates. "Someone could use this" is not hypothetical.

- **Screenshot-worthy output.** The morning briefing is human-readable text. Screenshot shows:
  agent output + audit log + delegation chain canvas all in one frame. Strongest possible YC
  artifact.

- **Moat is visible without explanation.** The delegation chain (orchestrator → gmail-agent →
  github-agent) appears as a live graph in the audit panel. A viewer sees "three boxes, three
  DPoP proofs, one human" and understands in 10 seconds what Auth0 cannot do.

## End-to-End User Journey

### Step 1 — First visit, signup
User lands on `briefcase.sharkauth.dev`. Enters email + password.
**Shark features:** `POST /auth/signup`, email verification flow (`/auth/verify-email`).
**SDK:** `@sharkauth/react` `useSignUp` hook → magic-link email delivered via Resend.

### Step 2 — Email verification
User clicks magic-link in email. Redirected back, session established.
**Shark features:** magic link verify (`GET /auth/magic-link/verify`).
**SDK:** `@sharkauth/react` `useAuth`, `magicLink.verify`.

### Step 3 — First login (subsequent visits)
Email + password login. Redirected to dashboard.
**Shark features:** `POST /auth/login`, server-side session cookie.
**SDK:** `useSignIn` hook.

### Step 4 — MFA enrollment
User prompted to add TOTP on first-login dashboard banner.
**Shark features:** `POST /auth/mfa/enroll`, `POST /auth/mfa/confirm`.
**SDK:** `useMFA` hook — show QR, confirm code, display recovery codes.

### Step 5 — Connect Gmail vault
User clicks "Connect Gmail." OAuth round-trip to Google. Shark stores token encrypted at rest.
**Shark features:** vault OAuth redirect, `GET /api/v1/vault/google_gmail/connect`,
token stored via FieldEncryptor.
**SDK:** Python backend initiates vault connect; TS frontend polls `vault.getConnection`.

### Step 6 — Connect GitHub vault
Same flow. OAuth to GitHub, Shark stores token.
**Shark features:** `GET /api/v1/vault/github/connect`.
**SDK:** same pattern.

### Step 7 — First agent task: "Run morning briefing"
User clicks "Generate Briefing." Backend spawns orchestrator agent.

Flow:
1. Python backend issues `client_credentials` token for `briefing-orchestrator` with DPoP.
2. Orchestrator calls Anthropic Claude (tool_use mode). Claude emits two tool calls:
   `fetch_gmail_summary` and `fetch_github_notifications`.
3. For each tool call, backend does token-exchange from orchestrator token → sub-agent token
   (narrower scope: `vault:gmail:read` or `vault:github:read`).
4. Sub-agent token used to call `GET /api/v1/vault/{provider}/token` with DPoP-bound request.
5. Sub-agent calls real Gmail API / GitHub API with retrieved token.
6. Results returned to Claude as tool outputs. Claude synthesises briefing.
7. Briefing displayed in frontend.

**Shark features:** DPoP token issuance (RFC 9449), token exchange (RFC 8693), `act` chain,
vault retrieval with jkt binding, `may_act` delegation policies.
**SDK Python:** `OAuthClient.get_token_with_dpop`, `OAuthClient.token_exchange`,
`VaultClient.fetch_vault_token`.
**SDK TypeScript:** `client.agents.get` to display agent status.

### Step 8 — View delegation chain in audit panel
User clicks "Audit" tab. Sees:
- `briefing-orchestrator` → `gmail-agent` (act chain depth 2)
- `briefing-orchestrator` → `github-agent` (act chain depth 2)
- Each hop: DPoP jkt, scope, timestamp.

**Shark features:** audit log, delegation chain canvas.
**SDK:** `client.audit.list`, TypeScript frontend renders chain graph.

### Step 9 — Draft reply agent (scope escalation attempt caught)
User selects a Gmail thread, clicks "Draft Reply." New agent `draft-agent` is spawned.
`draft-agent` attempts to call `vault:gmail:send` — but it was only delegated `vault:gmail:read`.
Shark returns `403 insufficient_scope`. Audit log shows the failed attempt.

**Shark features:** scope enforcement at delegation boundary, audit logging of 403.
**SDK:** `OAuthClient.token_exchange` with disallowed scope → server rejects.

### Step 10 — Revoke gmail-agent
User goes to Settings > Agents, clicks "Revoke gmail-agent."
All tokens for `gmail-agent` die. Next briefing run: gmail step fails gracefully, only GitHub
section renders.

**Shark features:** `POST /agents/{id}/tokens/revoke-all`.
**SDK Python:** `AgentsClient.revoke_agent_tokens`. TypeScript: `client.agents.update` to
show revoked state in UI.

### Step 11 — Disconnect Gmail vault
User clicks "Disconnect Gmail." Vault connection deleted. Any tokens derived from Gmail
vault connection cascade-revoked.

**Shark features:** `DELETE /api/v1/vault/connections/{id}`, cascade to derived tokens.
**SDK:** `VaultClient.delete_connection`.

### Step 12 — Session listing + revocation
User goes to Settings > Sessions. Sees all active sessions (devices/browsers). Clicks
"Sign out all other sessions."

**Shark features:** `GET /auth/sessions`, `DELETE /auth/sessions/{id}`.
**SDK:** `SessionsClient.list_sessions`, `SessionsClient.revoke_session`.

## Architecture

### Backend — FastAPI + Python (`shark_auth` SDK)

```
briefcase/
  backend/
    main.py                # FastAPI app, mounts routers
    routers/
      auth.py              # /api/auth/* — signup, login, MFA, session
      briefing.py          # /api/briefing — orchestrate agents, call LLM
      vault.py             # /api/vault — connect/disconnect Gmail + GitHub
      agents.py            # /api/agents — list, revoke
      audit.py             # /api/audit — fetch audit events
    agents/
      orchestrator.py      # briefing-orchestrator: DPoP token, LLM call, tool dispatch
      gmail_agent.py       # gmail-agent: token exchange + Gmail API
      github_agent.py      # github-agent: token exchange + GitHub API
      draft_agent.py       # draft-agent: token exchange + Gmail send (scoped narrow)
    shark.py               # shark_auth.Client singleton
    config.py              # env vars: SHARK_URL, SHARK_ADMIN_KEY, ANTHROPIC_API_KEY
```

Routes: `/api/auth/*`, `/api/briefing`, `/api/vault/*`, `/api/agents`, `/api/audit`.
Auth enforced via `shark_auth.session.validate_session` on every protected route.
No raw HTTP to Shark — all calls through Python SDK.

### Shark Instance

Self-hosted single binary, embedded SQLite. Hosted on Hetzner CX21 (2 vCPU, 4 GB RAM).
`shark serve` on boot. Admin key in env. No reverse proxy — SDK connects directly.

**Pre-registered agents (boot-time script `scripts/seed_agents.py`):**
- `briefing-orchestrator` — DPoP, `vault:gmail:read vault:github:read mcp:write`
- `gmail-agent` — DPoP, `vault:gmail:read`
- `github-agent` — DPoP, `vault:github:read`
- `draft-agent` — DPoP, `vault:gmail:read` (NOT send — intentional scope demo)

**`may_act` policies:**
- `briefing-orchestrator` → `gmail-agent` (scope: `vault:gmail:read`)
- `briefing-orchestrator` → `github-agent` (scope: `vault:github:read`)
- `briefing-orchestrator` → `draft-agent` (scope: `vault:gmail:read`)

### LLM Agents

Real production Anthropic API calls. `claude-sonnet-4-6` model. Tool-use mode.

Claude sees two tools:
```python
tools = [
    {"name": "fetch_gmail_summary", "description": "..."},
    {"name": "fetch_github_notifications", "description": "..."},
]
```

When Claude emits tool calls, backend performs token-exchange and real API calls, returns
results as tool outputs. Claude synthesises final briefing.

### Vault Providers

- **Gmail** — `gmail.readonly` scope. Real Google OAuth. Fetches last 10 unread threads.
- **GitHub** — `notifications` scope. Real GitHub OAuth. Fetches unread notifications.

## Real Prod Agents (Named + Scoped)

### `briefing-orchestrator`

- **LLM:** Anthropic Claude Sonnet 4.6 (tool_use)
- **Scope at registration:** `vault:gmail:read vault:github:read mcp:write`
- **DPoP lifecycle:** keypair generated at server boot via `DPoPProver.generate()`, `jkt`
  registered with Shark at agent creation. Key held in memory, regenerated on restart.
- **Delegation chain:** root agent (acts on behalf of user, `sub = user_id`)
- **Third-party APIs:** none directly — delegates to sub-agents for real API calls.
  Calls Anthropic API with tool outputs from sub-agents.

### `gmail-agent`

- **LLM:** none — pure API caller
- **Scope at registration:** `vault:gmail:read`
- **DPoP lifecycle:** own keypair, own `jkt`. Token-exchanged from orchestrator token.
- **Delegation chain:** `briefing-orchestrator` → `gmail-agent` (act chain depth 2)
- **Third-party API:** Gmail API `GET /gmail/v1/users/me/messages` + `GET /gmail/v1/users/me/threads`.
  Uses vault token retrieved via DPoP-bound `GET /api/v1/vault/google_gmail/token`.

### `github-agent`

- **LLM:** none
- **Scope at registration:** `vault:github:read`
- **DPoP lifecycle:** own keypair, own `jkt`.
- **Delegation chain:** `briefing-orchestrator` → `github-agent` (act chain depth 2)
- **Third-party API:** GitHub API `GET /notifications`. Token from vault.

### `draft-agent`

- **LLM:** Anthropic Claude Sonnet 4.6 (for draft generation)
- **Scope at registration:** `vault:gmail:read` — intentionally missing `vault:gmail:send`
- **DPoP lifecycle:** own keypair per user session request.
- **Delegation chain:** `briefing-orchestrator` → `draft-agent`
- **Third-party API:** would call Gmail send API — but scope enforcement prevents it.
  This is the **intentional failure demo** — scope escalation attempt caught by Shark.
- **What it demonstrates:** sub-agent cannot exceed parent's delegated scope.

## Agent Flows Dogfooded

1. **Agent registration** — `seed_agents.py` runs at Shark boot. Registers 4 agents, configures
   `may_act` policies. Admin dashboard shows all 4 with their `client_id` + DPoP `jkt`.

2. **DPoP-bound client_credentials** — `briefing-orchestrator` calls
   `OAuthClient.get_token_with_dpop(grant_type="client_credentials", ...)` at request time.
   Token has `cnf.jkt` bound to orchestrator's keypair. Audit: `oauth.token.issued`.

3. **Orchestrator dispatches LLM task** — Claude receives tools, emits `fetch_gmail_summary`.
   Backend intercepts tool call, runs token exchange.

4. **Token exchange (depth 2)** — `OAuthClient.token_exchange(subject_token=orch_token,
   scope="vault:gmail:read", dpop_prover=gmail_prover)`. Shark mints new token:
   - `sub = user_id` (unchanged — still acting for the user)
   - `act.sub = briefing-orchestrator`
   - `scope = vault:gmail:read`
   - `cnf.jkt = gmail_agent_jkt`
   Audit: `oauth.token.exchanged`.

5. **Vault retrieval with DPoP** — `gmail-agent` calls
   `VaultClient.fetch_vault_token("google_gmail", dpop_prover=gmail_prover)`.
   Shark validates: DPoP proof matches `cnf.jkt`, scope contains `vault:gmail:read`,
   `sub` maps to the vault connection owner. Returns Gmail access token.
   Audit: `vault.token.retrieved` with `act_chain` in metadata.

6. **Real Gmail API call** — `gmail-agent` calls Gmail with the retrieved token.
   Returns thread summaries to Claude as tool output.

7. **Sub-agent scope enforcement** — `draft-agent` token-exchanged with
   `scope="vault:gmail:send"`. Shark rejects: parent's delegated scope is `vault:gmail:read`,
   cannot escalate. Returns `403 insufficient_scope`. Logged as `oauth.token.exchange.denied`.

8. **Audit log shows full delegation chain** — frontend queries `AuditClient.list_events`.
   Each event has `actor_id`, `act_chain` (JSON array of parent agent IDs), `target`, timestamp.

9. **User revokes gmail-agent** — `AgentsClient.revoke_agent_tokens("gmail-agent-id")`.
   All gmail-agent tokens invalidated. Next briefing request: vault retrieval returns 401.
   Frontend shows "Gmail disconnected from agent" notice.

10. **Cascade via user revocation** — `UsersClient.cascade_revoke_agents(user_id)`. All 4
    agents' tokens revoked in one call. Audit shows 4 `agent.tokens.revoked` events.

## Vault Flows Dogfooded

- **Connect Gmail:** OAuth round-trip via Shark vault connect. Token stored encrypted at rest.
- **Connect GitHub:** same.
- **Retrieve Gmail token (agent):** DPoP-bound `VaultClient.fetch_vault_token` validates chain.
- **Retrieve GitHub token (agent):** same, scoped to `vault:github:read`.
- **Disconnect Gmail:** `VaultClient.delete_connection("google_gmail")`. All tokens derived
  from this connection cascade-revoked. GitHub connection and its agent tokens survive.
- **Verify cascade:** after Gmail disconnect, `github-agent` still works. `gmail-agent`
  returns 401 on vault fetch. Demonstrates precision revocation.

## Failure Flows to Demonstrate

### 1. Scope escalation blocked
`draft-agent` requests `vault:gmail:send` via token exchange. Parent's scope is `vault:gmail:read`.
Shark returns `403 insufficient_scope`. Audit logs `oauth.token.exchange.denied` with
`requested_scope` + `allowed_scope` in metadata. Frontend shows "Agent tried to escalate scope
— blocked by Shark." Red entry in audit timeline.

### 2. DPoP jkt mismatch
Attacker intercepts `gmail-agent` token, attempts vault retrieval with WRONG keypair.
Shark validates DPoP proof against `cnf.jkt` in token. Mismatch → `401 invalid_dpop_proof`.
Demonstrated via `seed_agents.py` attack-mode flag (optional screencast scene).

### 3. Revocation during in-flight request
User clicks "Revoke gmail-agent" mid-briefing. The in-flight Gmail vault retrieval returns 401.
Backend handles gracefully: briefing renders GitHub section only, notes Gmail was revoked.
Demonstrates real-time revocation propagation.

## Stack Choices + Rationale

| Layer | Choice | Rationale |
|---|---|---|
| Python backend | FastAPI 0.115 + uvicorn | Mirrors target integrator stack (MCP server builders). `shark_auth` Python SDK is primary SDK. Demo-api IS the integration tutorial. |
| Frontend | Next.js 15 (App Router) | SSR cookie forwarding works natively with `@sharkauth/next`. Server components + middleware = clean auth pattern. Target customers build on Next.js. |
| LLM | Anthropic SDK (`anthropic` 0.25+) | Claude Sonnet 4.6 is the model powering this very tool. Brand alignment. Tool-use is stable and well-documented. |
| Shark instance | Single binary, embedded SQLite | The whole point. Demo proves `shark serve` = zero ops. CX21 Hetzner ($5/mo) is sufficient. |
| Hosting | Hetzner VPS (Shark) + Fly.io (FastAPI) + Vercel (Next.js) | Shark on bare metal to prove single-binary story. FastAPI on Fly.io for instant deploys. Vercel for frontend CDN. Total cost < $20/mo. |
| Gmail + GitHub | Real OAuth, read-only scopes | Verified providers per v0.1 audit. No mocking. Real tokens, real data. |

## Risks + Mitigations

| Risk | Mitigation |
|---|---|
| **LLM rate limits during demo recording** | Pre-cache a briefing result as JSON fixture. For recording: use fixture unless live call succeeds in < 5s. Flag with comment in code. |
| **Gmail OAuth flakes mid-recording** | Pre-record OAuth connect flow separately. Splice into main screencast. Or seed vault connection directly via `POST /admin/vault/connections/seed-demo`. |
| **DPoP proof nonce errors from Shark** | Backend retries once on `use_dpop_nonce` error (RFC 9449 §4.3). Log the nonce, retry with it. Already handled in Python SDK `DPoPProver`. |
| **Demo data pollution across takes** | `scripts/reset_demo.py` — deletes user + all agents + vault connections + audit events for `demo@briefcase.dev`. Runs between recording takes. 3 seconds. |
| **Python SDK method gaps** | Audit SDK coverage before build starts. If method missing, add it to `sdk/python/shark_auth/` first. Don't raw-HTTP in demo code. |
| **Claude tool-use schema rejected** | Keep tool schemas simple (no nested required fields). Test with `claude-3-5-sonnet-20241022` first if 4.6 has issues. |
| **Fly.io cold start adds latency to first briefing** | Keep backend warm with UptimeRobot ping every 5 min. `fly machine start --auto-start` on deploy. |
| **GitHub API returns empty notifications** | Seed a test org with unread notifications via GitHub personal account. Or mock GitHub section with real token but fake data fixture. |

- Login screen. Enter email + password. MFA prompt appears. Enter TOTP code. Dashboard loads.
- Caption overlay: "Human auth: OAuth 2.1 + TOTP. Same as any auth platform."

**Scene 2 (0:20-0:45) — Connect vault**
- Click "Connect Gmail." Google OAuth screen. Authorize. Back to dashboard. Gmail card shows
  connected. Click "Connect GitHub." Same. Two vault cards show green status.
- Caption: "Vault: Gmail + GitHub tokens stored encrypted. Agents will request them, not you."

**Scene 3 (0:45-1:45) — Morning briefing runs**
- Click "Generate Briefing." Spinner. Cut to Shark admin dashboard — delegation chain canvas
  updating live. Three boxes: orchestrator → gmail-agent → github-agent. Each box gets a
  green checkmark as it completes. Briefing text appears.
- **THIS IS THE HERO SHOT.** Split screen: briefing text left, delegation chain right.
- Caption: "Orchestrator delegates to sub-agents. Each hop: different DPoP keypair, narrower scope."

**Scene 4 (1:45-2:15) — Audit log**
- Click "Audit" tab. Show timeline: `oauth.token.issued` (orchestrator), two
  `oauth.token.exchanged` entries (gmail/github agents), two `vault.token.retrieved` entries.
  Hover one: full `act_chain` JSON visible.
- Caption: "Every hop audited. `act` claim shows the full chain back to the human."

**Scene 5 (2:15-2:40) — Scope enforcement**
- Click "Draft Reply" on a Gmail thread. Error banner: "gmail-agent cannot send — scope
  insufficient." Audit log shows red `oauth.token.exchange.denied` entry.
- Caption: "Sub-agents cannot escalate scope. Shark enforces at the delegation boundary."

**Scene 6 (2:40-3:10) — Revocation**
- Settings > Agents. Click "Revoke gmail-agent." Confirmation. Agent card turns red. Run
  briefing again. Only GitHub section renders. Gmail section: "Agent revoked."
- Caption: "Revoke one agent. Others survive. Blast radius is surgical."

**Scene 7 (3:10-3:20) — Close**
- `shark serve` in terminal. One binary. One command.
- Caption: "SharkAuth. Auth for humans and the agents they ship."
